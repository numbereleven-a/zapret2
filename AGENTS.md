# AGENTS.md — статический аудит nfq2/ и lua/ (zapret2-master)

## 0) Цель
Провести полный статический анализ реализации подсистем:
- zapret2-master/nfq2/
- zapret2-master/lua/

и найти реальные дефекты/несоответствия в коде на реальных входах из сети (nfqueue/pcap/пакеты/байтовые потоки/Lua-хуки), а не на маловероятных пользовательских вводах.

Manual (zapret2-master/docs/manual.en.md) использовать как дополнительный источник ожидаемого поведения, но НЕ как “закон” и НЕ как фильтр на находки.

Язык всех артефактов отчёта: русский.

---

## 1) Источник правды и условие завершения (обязательно)
Источник правды о прогрессе — файл:
- audit/coverage.csv со столбцами: path,status,reason,severity_counts

Где:
- status ∈ pending | analyzed_full | analyzed_partial | skipped

Работа считается завершённой ТОЛЬКО когда:
- в audit/coverage.csv нет строк со статусом pending
- и каждая строка из audit/SCOPE_LIST.txt присутствует в coverage.csv ровно один раз

В чат выводить только прогресс (см. раздел 12).

---

## 2) Приоритет критериев (manual не “закон”)
Manual НЕ является “законом” и НЕ является фильтром на находки.

Находка валидна даже если manual её не упоминает, если выполнено одно из:
- ошибка обработки сетевых/пакетных данных (границы, парсинг, инварианты)
- ошибка обработки системных вызовов/IO (errno, частичные операции read/write/send/recv, ресурсы, nfqueue/pcap API)
- контрактная ошибка между модулями (включая nfq2 ↔ lua: структуры/форматы/коды ошибок/инварианты)
- недостижимость/ложная уверенность из-за флагов/инициализации
- утечки/двойные освобождения/переполнения/UB, если достижимо на реальном сетевом трафике

Manual использовать ТОЛЬКО как:
- источник ожидаемых режимов/потоков/конфигов/скриптов
- подтверждение несоответствия, если оно есть
Отсутствие требования в manual НЕ является доказательством корректности.

---

## 3) Scope и исключения (строго)
Анализировать ТОЛЬКО файлы внутри:
- zapret2-master/nfq2/
- zapret2-master/lua/

Исключить полностью (не читать/не упоминать в находках):
- любые *.sh и любые shell-скрипты
- nfq2/gzip.c, nfq2/gzip.h и вообще всё, что связано с gzip (как тема/логика/обсуждение)
- nfq2/Makefile, nfq2/BSDmakefile
- любые изменения/анализ, связанные с build/setup
- любые изменения/анализ, связанные с signal handlers (signal(), sigaction(), SIG* и завязанная логика)

Если файл бинарный/ресурсный (ico/o/a/obj и т.п.) — ставить skipped с причиной "binary resource", без анализа.

## 4) Жёсткий игнор “известных пунктов”
Если находка совпадает с любым пунктом ниже — СРАЗУ ПРОПУСКАТЬ:
- не анализировать и не подтверждать
- не добавлять в triage
- не делать карточку
- не предлагать фиксы/патчи
- не делать дополнительных поисков по репо ради этого пункта

Исключение: если рядом/в том же месте обнаружен ДРУГОЙ корень проблемы, который НЕ является частью известного пункта — анализировать только этот новый корень (не затрагивая известный).

Список “жёсткий игнор”:

- has_bad_ws_options() всегда возвращает 1 — has_bad_ws_options() (nfq2/*)
- AES-CTR: 64-битные load/store через uint64_t* (alignment)
- rtrim() — UB на пустой строке — rtrim() (nfq2/*)
- wgobfs() несовместима с orchestration-режимом по cutoff-логике — wgobfs() в lua/
- luaexec() не работает на Lua 5.1 из-за load() — lua/zapret-lib.lua: luaexec() / load()
- HttpPos(PM_HTTP_METHOD) не поддерживает методы > 7 символов — HttpPos / PM_HTTP_METHOD (nfq2/*)
- Lua memcpy() пишет в неизменяемую строку → UB/краш — memcpy() в lua/ (или C↔Lua биндинг)
- trimstr / interlocked_get_dom: trimstr сразу после fgets — interlocked_get_dom + trimstr + fgets
- Нарушение контракта ctx в оркестрации — plan_instance_execute() / orchestrate(); circular(); nfq2/lua.c: raw_packet(); pcap()
- Unchecked return clock_gettime() в Lua-модулях — clock_gettime(); math.randomseed(...); pcap_write_packet()
- Потеря “хорошего” hostlist/ipset при неуспешном hot-reload — LoadHostList
- Тихая порча списков при длинных строках (256-байтный буфер + fgets) — AppendHostList
- circular() использует # на хэш-таблице — count_strategies: uniq[...] = true + if n~=#uniq then error(...)
- check_pos_cutoff и b_cutoff_all — Lua отсекается без conntrack (pos==NULL)
- ConntrackPoolPurge — системные вызовы: clock_gettime в очистке conntrack
- Несоответствие manual по hostname_is_ip, lua_pushf_ctrack
- Несогласованный контракт ctx при оркестрации Lua
- NULL hostname в ConntrackPoolDump
- Некорректный winsize_calc на первом пакете со scale
- Несостыкованный API csum_partial

## 6) Что искать (критерии)
Искать проблемы только по nfq2/ и lua/:

- заявлено/вызывается, но фактически no-op или всегда падает
- недостижимые ветки (условия/флаги/инициализация)
- несоответствие контрактов между модулями, включая nfq2 ↔ lua
- неверная обработка системных вызовов/IO (errno, частичные операции, ресурсы, nfqueue API, pcap и т.д.)
- ошибки обработки сетевых данных: длины/оффсеты/границы/переполнения/подписи типов/NUL-терминация
- несоответствие реализации manual в части nfq2/lua (вторичный приоритет)

---

## 7) Минимальный чеклист на каждый C/Lua файл (обязательно)
Для каждого файла выполнить минимальный чеклист и явно отметить результат. Если неприменимо — написать "N/A" и почему.

1) IO/системные вызовы:
- проверка кодов возврата и errno
- проверка частичных операций (read/write/send/recv) и корректной докрутки/ретраев (если требуется контрактом)

2) Границы и парсинг сетевых данных:
- проверка длины/оффсетов/диапазонов
- отсутствие переполнений size_t/int, неверных кастов/знаковости
- корректная NUL-терминация строк, если используется строковый API

3) Инварианты и контракты:
- что функция обещает вернуть/изменить
- проверка, что вызывающие стороны это соблюдают
- nfq2↔lua: форматы данных, коды ошибок, ожидания по ctx/структурам (в рамках scope)

4) Ошибочные ветки и ресурсы:
- освобождение ресурсов на всех путях ошибок
- отсутствие double free/use-after-free при достижимых сценариях

## 8) Политика “стороннего кода”
Если файл попал в SCOPE_LIST — он анализируется наравне с остальными.

Запрещены общие дисклеймеры:
- "полная верификация стороннего кода не проводилась"
- "сторонний код не проверялся"

Если есть ограничение, оно должно быть КОНКРЕТНЫМ (что именно не проверено и почему), и не должно противоречить scope.
Допустимый пример ограничения: "не анализировались исключённые темы: signal handlers".

---

## 9) Статусы analyzed_full / analyzed_partial (жёстко)
analyzed_full ставить только если:
- файл прочитан целиком
- в findings есть ссылки на несколько ключевых участков (функции/ветки) + сниппеты + примерные диапазоны строк
- минимальный чеклист (раздел 7) заполнен (N/A допускается с причиной)

analyzed_partial ставить только если:
- явно указано что именно не дочитано/не доказано (конкретные функции/блоки)
- причина конкретная (например: "не удалось доказать достижимость X без трассы", "остался неразобранный блок Y")
- минимальный чеклист заполнен насколько возможно (что не выполнено — тоже конкретно)

Запрещено писать "частичный обзор" без конкретики.

skipped — только по причине:
- исключение по правилам (например *.sh, gzip, makefile) или
- бинарный/ресурсный файл

## 10) Формат per-file результатов (обязательно)
Для каждого обработанного файла создавать/обновлять:
- audit/findings/<original_path>.md
(зеркалировать структуру директорий внутри audit/findings/)

Каждый findings-файл обязан содержать:

### A) Что проверено (конкретно)
- Какие сетевые/пакетные входы обрабатывает этот файл и где
- Какие ключевые функции/ветки разобраны (список)
- Какие контракты (форматы/коды ошибок/инварианты) проверены
- Минимальный чеклист (раздел 7): результат по пунктам (OK/Issue/N/A)
- Где использован manual (только релевантные места), если использован

### B) Найденные проблемы (обязательно по всем severity)
- Critical: (список или "нет")
- High: (список или "нет")
- Medium: (список или "нет")
- Low: (список или "нет")
- Info: (список или "нет")

Для каждой проблемы — шаблон:
- Где: файл → функция/символ → примерные строки → короткий сниппет
- Как задумано: по manual / месту вызова / контракту
- Что в коде сейчас: фактическое поведение (условия/данные/ветка)
- Итог: корень + последствия (безопасность/устойчивость/корректность)
- Доказательство достижимости: почему реально выполняется (или почему никогда)
- Как подтвердить: минимальная трасса/лог/тест/пример пакета
- Фикс: что конкретно поменять и где
