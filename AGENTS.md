# AGENTS.md — автономный статический аудит nfq2/ и lua/ (zapret2-master)

## 0) Цель
Автономно (без ожидания дополнительных сообщений пользователя) выполнить полный статический анализ:
- zapret2-master/nfq2/
- zapret2-master/lua/

и найти реальные дефекты на реальных входах из сети (nfqueue/pcap/пакеты/байтовые потоки/Lua-хуки), а не на маловероятных пользовательских вводах.

Manual (zapret2-master/docs/manual.en.md) использовать как дополнительный источник ожиданий, но НЕ как “закон” и НЕ как фильтр на находки.

Язык всех артефактов: русский.

---

## 1) Источник правды и условие завершения (обязательно)
Источник правды прогресса — файл:
- audit/coverage.csv со столбцами: path,status,reason,severity_counts

Где status ∈ pending | analyzed_full | analyzed_partial | skipped.

Работа считается завершённой ТОЛЬКО когда:
- в audit/coverage.csv нет строк со статусом pending
- каждая строка из audit/SCOPE_LIST.txt присутствует в coverage.csv ровно один раз

---

## 2) Приоритет критериев (manual не “закон”)
Manual НЕ является фильтром на находки.

Находка валидна, даже если manual её не упоминает, если есть:
- ошибка обработки сетевых/пакетных данных (границы, парсинг, инварианты)
- ошибка обработки системных вызовов/IO (errno, частичные операции, ресурсы, nfqueue/pcap API)
- контрактная ошибка между модулями (включая nfq2 ↔ lua)
- недостижимость/ложная уверенность из-за флагов/инициализации
- утечки/двойные освобождения/переполнения/UB, если достижимо на реальном трафике

Manual использовать только как:
- источник ожидаемого поведения/режимов/потоков
- подтверждение несоответствия, если оно есть

## 3) Scope и исключения (строго)
Анализировать ТОЛЬКО файлы внутри:
- zapret2-master/nfq2/
- zapret2-master/lua/

Исключить полностью (не читать/не упоминать):
- любые *.sh и shell-скрипты
- nfq2/gzip.c, nfq2/gzip.h и любые обсуждения gzip
- nfq2/Makefile, nfq2/BSDmakefile
- любые изменения/анализ, связанные с build/setup
- любые изменения/анализ, связанные с signal handlers (signal/sigaction/SIG* и завязанная логика)

Бинарные/ресурсные файлы (ico/o/a/obj и т.п.) → skipped с причиной "binary resource".

---

## 4) Жёсткий игнор “известных пунктов”
Если находка совпадает с любым пунктом ниже — сразу пропускать (не упоминать, не triage, не фиксить, не искать):
- has_bad_ws_options() всегда возвращает 1 — has_bad_ws_options() (nfq2/*)
- AES-CTR: 64-битные load/store через uint64_t* (alignment)
- rtrim() — UB на пустой строке — rtrim() (nfq2/*)
- wgobfs() несовместима с orchestration-режимом по cutoff-логике — wgobfs() в lua/
- luaexec() не работает на Lua 5.1 из-за load() — lua/zapret-lib.lua: luaexec() / load()
- HttpPos(PM_HTTP_METHOD) не поддерживает методы > 7 символов — HttpPos / PM_HTTP_METHOD (nfq2/*)
- Lua memcpy() пишет в неизменяемую строку → UB/краш — memcpy() в lua/ (или C↔Lua биндинг)
- trimstr / interlocked_get_dom: trimstr сразу после fgets — interlocked_get_dom + trimstr + fgets
- Нарушение контракта ctx в оркестрации — plan_instance_execute() / orchestrate(); circular(); nfq2/lua.c: raw_packet(); pcap()
- Unchecked return clock_gettime() в Lua-модулях — clock_gettime(); math.randomseed(...); pcap_write_packet()
- Потеря “хорошего” hostlist/ipset при неуспешном hot-reload — LoadHostList
- Тихая порча списков при длинных строках (256-байтный буфер + fgets) — AppendHostList
- circular() использует # на хэш-таблице — count_strategies: uniq[...] = true + if n~=#uniq then error(...)
- check_pos_cutoff и b_cutoff_all — Lua отсекается без conntrack (pos==NULL)
- ConntrackPoolPurge — системные вызовы: clock_gettime в очистке conntrack
- Несоответствие manual по hostname_is_ip, lua_pushf_ctrack
- Несогласованный контракт ctx при оркестрации Lua
- NULL hostname в ConntrackPoolDump
- Некорректный winsize_calc на первом пакете со scale
- Несостыкованный API csum_partial

Исключение: если рядом обнаружен другой корень проблемы, не совпадающий с пунктами выше — анализировать только новый корень.

## 6) Минимальный чеклист на каждый файл (обязательно)
Для каждого файла выполнить и явно отметить (OK/Issue/N/A + почему):

1) IO/системные вызовы:
- проверка кодов возврата/errno
- частичные операции read/write/send/recv (если применимо)

2) Границы и парсинг сетевых данных:
- длины/оффсеты/диапазоны
- переполнения size_t/int, знаковость
- NUL-терминация строк при строковом API

3) Инварианты/контракты:
- что функция обещает и что реально делает
- соблюдение контрактов вызывающей стороной
- nfq2↔lua контракты в рамках scope

4) Ошибочные ветки/ресурсы:
- ресурсы освобождаются на всех путях
- нет достижимых double free/UAF

## 7) Политика “стороннего кода”
Если файл в SCOPE_LIST — анализировать наравне с остальными.
Запрещены общие дисклеймеры вроде "полная верификация стороннего кода не проводилась".
Если есть ограничение — только конкретное (что именно не проверено и почему) и в рамках scope.

---

## 8) Статусы analyzed_full / analyzed_partial (жёстко)
analyzed_full:
- файл прочитан целиком
- в findings есть несколько ключевых участков + сниппеты + примерные строки
- чеклист заполнен (N/A допускается с причиной)
- заполнены все severity-блоки (Critical/High/Medium/Low/Info)

analyzed_partial:
- конкретно указано, что не дочитано/не доказано (функции/блоки)
- конкретная причина
- чеклист заполнен насколько возможно

skipped:
- исключение по правилам или бинарный ресурс

## 9) Формат per-file результатов (обязательно)
Для каждого файла создавать/обновлять:
- audit/findings/<original_path>.md
(зеркалировать структуру директорий внутри audit/findings/)

Каждый findings-файл содержит:

A) Что проверено (конкретно)
- какие сетевые/пакетные входы и где
- какие функции/ветки разобраны
- какие контракты проверены
- чеклист (OK/Issue/N/A)
- где использован manual (если использован)

B) Найденные проблемы (обязательно по всем severity)
- Critical:
- High:
- Medium:
- Low:
- Info:

Каждая проблема по шаблону:
- Где (файл/функция/строки/сниппет)
- Как задумано (manual/контракт/место вызова)
- Что в коде сейчас
- Итог (корень + последствия)
- Доказательство достижимости
- Как подтвердить
- Фикс (конкретно)

## 11) Построение SCOPE_LIST (источник правды)
Перед анализом создать:
- audit/SCOPE_LIST.txt (полный отсортированный список файлов в nfq2/ и lua/)

Исключить из списка:
- *.sh
- nfq2/gzip.c, nfq2/gzip.h
- nfq2/Makefile, nfq2/BSDmakefile

Затем создать/синхронизировать:
- audit/coverage.csv: каждая строка SCOPE_LIST должна быть в coverage.csv с status=pending (если ещё не обработано)

---

## 12) Правило непрерывного прогресса (анти-early-stop, автономно)
Если в audit/coverage.csv есть pending, запрещено завершать выполнение после обработки одного файла.

За один запуск необходимо:
- обработать минимум 25 pending-файлов ИЛИ
- дойти до pending_left=0
что наступит раньше.

Если выполнение нужно прервать из-за лимитов/ограничений среды:
- обязательно обновить audit/coverage.csv по всем уже обработанным файлам
- создать audit/next_pending.txt со строками:
  - next_pending=<path>
  - pending_left=<N>
  - last_processed=<path>
- собрать промежуточный отчёт audit/report.md (даже если не финальный), чтобы прогресс был полезен без чата

---

## 13) Вывод в чат (минимум)
В чат выводить только прогресс-строки:
- <path> → <status> → counts(C/H/M/L/I) → pending_left=<N>

Никаких длинных отчётов в чат.
